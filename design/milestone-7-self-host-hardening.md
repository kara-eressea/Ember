# Milestone 7 — Self-Host Hardening

**Goal:** everything required to responsibly run a personal EmberChat instance on the public internet, and to make self-hosting by others excellent — plus the design pass for the standalone desktop client. Rescoped 2026-07-16 from "public-service hardening" when the tenancy decision changed (decisions.md §2): EmberChat is self-hostable software with admin-only instances, not a managed service.

**Depends on:** Milestones 1–2 minimum; realistically follows 1–6.

## Scope

- **Admin-only accounts:** registration endpoint disabled (config-gated off by default); an admin CLI (or bootstrap env mechanism) creates accounts and resets passwords. No email flows in v1.0 — the Brevo/mailer plan is mooted (decisions.md §5). Removing the open-registration path also closes the register-endpoint enumeration item from the server hardening backlog.
- **Exposure hardening** (the instance is internet-reachable even with one user): account lockout/backoff on failed logins, per-route rate limits kept honest, `@fastify/helmet`-style headers, WS origin checks, CSRF posture review, secure cookie flags, dependency audit, secrets-handling review. Fold in the standing **server hardening backlog** (expired `auth_sessions` cleanup + session cap, gateway connect/hello rate-gate, monotonic-clock note).
- **Self-host story — setup must be fairly painless:** a copy-paste-quality example `compose.yaml` + `.env` (server + web + Postgres, TLS via reverse proxy) that works with minimal edits, a written deploy guide (`docs/self-hosting.md`: prerequisites, env reference, reverse-proxy examples, upgrade path), automated Postgres backups + documented restore drill, `.env.example` kept exhaustive.
- **Image tags & upgrade safety:** publish ghcr images as `latest`, per-version (`vX.Y.Z` + rolling `vX.Y`/`vX`), and `pre-release`/`edge` tags so self-hosters can pin their risk appetite. **Upgrades are compatible by default; when one isn't, the app raises a massive flag instead of proceeding** — a naive `docker pull` must never ruin a database: schema migrations run forward-only on boot behind a startup gate that refuses to start (with a loud, actionable error naming the release notes and the backup step) when the jump crosses a flagged breaking migration or the database is *newer* than the binary (accidental downgrade). Breaking releases are always a major version bump with explicit upgrade notes.
- **Version visibility & update check:** the running version (git tag/semver, baked at image build) is shown in the app UI (settings/about + a `/healthz`-style endpoint); a **daily update check** against the GitHub Releases API surfaces a quiet "update available" hint in the UI — with a config switch to disable the phone-home for people who'd rather not (the check reveals the instance's existence to GitHub, nothing more).
- **Staff reporting (SFC):** the "Alert Staff" flow from the official client — report a user/channel to F-List staff with recent log context attached (M6 parity-audit decision; tenancy-independent).
- **Retention & logs:** extend the `RETENTION_POLICY` enum beyond `forever` (30d/90d/1yr/Forever) enforced by the M2 retention job; streaming log export (M5 backlog item); failed/pending outbox TTL sweep (M4 backlog item); cursor-path per-sub replay budget (M2 backlog item).
- **Ops:** structured logging (pino) + health endpoints; alerting on session-engine error rates and reconnect storms is best-effort at single-user scale; deploy runbook acknowledging that restarts log the user out of F-Chat. (OpenTelemetry and messages-table partitioning are dropped to a wishlist note — no scale to justify them.)
- **Credential custody:** session-only in-memory credentials remain the default (decisions.md §3). The **full bouncer model (restart-surviving sessions) is an eventual goal**: when it's time, encrypted at-rest storage lands as an explicit per-user **opt-in**, preceded by the F-List outreach decision. Single-tenancy softens the custody question (the operator is the user) but doesn't dissolve it — the desktop client dissolves it.
- **Standalone-split design pass (design only, no implementation):** a written design (`design/standalone-client.md`) for the eventual desktop app (Tauri/Electron — decided 2026-07-16). Decide: what extracts into a shared session library (`fchat-session`, rate gate, ticket manager vs. their Drizzle/gateway couplings); embedded-bouncer vs. direct-connect architecture (M8's bundled-server variant is the starting sketch); storage story (SQLite/pglite vs. embedded Postgres); what the gateway protocol looks like in-process; Tauri vs. Electron evaluation. Goal: v1.0 code stops accruing coupling that makes the split harder — the design names the boundaries even though the extraction waits.

## Codebase sanity check vs. the pivot (2026-07-16)

Pre-kickoff sweep of the implemented codebase against the new tenancy model. Verdict: nothing needs pre-M7 surgery, but the findings bind M7's plan:

- **Sequencing constraint (load-bearing):** there is no seed script, bootstrap, or admin CLI anywhere — every account in dev, integration tests, and E2E is created by driving the public `/register` endpoint (`apps/web/e2e/helpers.ts` `registerAndConnect()`, `global-setup.ts`, the auth specs, many server tests). Gating registration off must land **in the same step** as the admin CLI + test-helper migration, or a fresh instance is unenterable and the suite breaks.
- **Conflicting surfaces to remove/fix with that step:** `POST /register` (`auth/routes.ts`) has no config gate; the Register UI (`Register.tsx`, `/register` route, store/api actions); "Create account" links on Landing + Login; the Register form's ToS checkbox (no ToS exists — dies with the form); the Login "Forgot?" tooltip promising email-based reset in v1.0 (now false — reset is an admin-CLI op).
- **Cleanup riders:** "suits a public deployment" comments on `RATE_LIMIT_MAX`/`AUTH_RATE_LIMIT_MAX`; the unused `app_users.emailVerifiedAt` column and the "email_tokens arrive with M7" schema comment; M8's stale "renderer points at the hosted service" line. Note `app_users.email` is `notNull().unique()` — the admin CLI still supplies an email even though none is ever sent.
- **Confirmed non-issues:** zero mailer/SMTP/Brevo/Mailpit code, config, or deps; no ToS/privacy/CAPTCHA/abuse-intake surfaces ever built; README doesn't market a hosted service.
- **Standalone-split posture (good news):** the session engine is already almost a library — `fchat-session`, event bus, rate gate, session state, registry, both `flist-api` files, and the credential vault import only `ws`/node stdlib/`fchat-protocol`; all coupling is inverted through the session event bus (history/gateway/directory subscribe in; the engine never calls out). Exactly one file touches Postgres — `connect-identity.ts`, an app-level orchestration helper that stays behind in `apps/server` on extraction. Trivial cleanups: the vault is misfiled under `flist-accounts/`; `ticket-manager.ts`'s type-only vault import moves with it. The design pass starts from "mostly done".

## Verification

- Integration: lockout/backoff behavior; rate-limit headers; registration path confirmed unreachable when disabled; admin CLI account-create/reset round-trip.
- Restore drill: backup → wipe → restore → data intact; user can unlock and reconnect.
- Deploy guide walked end-to-end on a clean VM/container host (the guide is the test script) — starting from the example `compose.yaml`/`.env` with minimal edits.
- Upgrade gate: simulated version jump across a flagged breaking migration refuses to boot with the loud error; normal minor upgrade migrates cleanly; binary-older-than-database refuses.
- Version/update surface: UI shows the built version; update check flags a newer mock release and stays silent when current (and stays off when disabled).
- SFC report round-trip against fchat-sim.
- External review pass (even informal) of the credential-vault handling (no persistence, no logging, cleared on disconnect) before calling v1.0.
